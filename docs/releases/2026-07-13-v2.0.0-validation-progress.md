## Findings

### V-1 (informational) — npm package name collision, not a release blocker

Root `package.json` (`token-optimizer-mcp`, unscoped, not `private`) matches
an unrelated published package (`ooples/token-optimizer-mcp`, currently
`2.17.0`). Confirmed this is cosmetic: `.github/workflows/publish-npm.yml`
only ever publishes `packages/installer` (`@softawarest/token-optimizer-installer`,
correctly scoped). No action taken per maintainer decision.

### V-2 (fixed) — `repair --json` / `uninstall --json` silently no-opped real operations

`packages/installer/bin/token-optimizer.js` treated `args.json === true` as
equivalent to `args["dry-run"] === true` for the `repair`/`uninstall`
branch: `uninstall --json` (no `--dry-run`) printed a full operations plan,
exited 0, and reported `77` operations — but **applied nothing**. Verified
by installing into a scratch `HOME`, running `uninstall --json`, and
confirming the manifest, LaunchAgent plist, and client caches were still on
disk afterward. `install`'s dry-run branch never had this bug (it only
checks `--dry-run`). No existing test exercised this path — the only prior
`uninstall --json` test ran it on an already-uninstalled state, which
short-circuits before the buggy branch.

Fixed: the branch now only short-circuits on `--dry-run`; `--json` only
changes output formatting, matching `install`'s existing behavior. Verified
the fix: reinstalled into a fresh scratch `HOME`, ran `uninstall --json`,
confirmed the manifest and plist were actually removed. Added a regression
test (`test/installer/cli.test.ts`, "uninstall --json without --dry-run
actually applies, not just previews"). Updated
`packages/installer/README.md` to describe `--json` as an output-format
flag, not a preview flag. Full suite: 372/372 passing after the fix.

### V-3 (informational) — local LLM gateway unreachable in this environment

`check_local_llm_health` and `run_test_verdict` both reported
`401 Unauthorized` from the configured gateway
(`https://llm-proxy.lnf.gr/v1`). Command execution and exit-code handling
remained correct and authoritative regardless (by design — see
`docs/security/threat-model.md` and this repo's `CLAUDE.md`), and the tool
degraded to `"uncertain"` rather than fabricating a confident verdict, which
matches the documented conservative-fallback contract. Flagging because this
means no real LLM-triage path was exercised this session — only the
deterministic command path. Gateway credential health is outside this
session's authority to fix.

### V-4 (open, not fixed this session) — Python auto-detection misses pyproject.toml-only projects

`src/detector.ts`'s Python branch only fires on `pytest.ini`, `conftest.py`,
`requirements.txt`, or `Pipfile`. A `pyproject.toml`-only project (Poetry,
uv, hatch, PEP 621 — the current standard for new Python projects) is never
detected; `detectCommands` returns `[]`, and `run_test_verdict` reports "No
tests or build tools detected" even with a real test suite present.
Confirmed with a real fixture workspace and the actual MCP tool call.
Flagged as a background task (not implemented in this session) rather than
fixed inline, since it changes detection behavior for a whole ecosystem and
deserves its own test coverage and doc update per this repo's `CLAUDE.md`.

### V-5 (open, not fixed this session) — possible marketplace-cache residue after uninstall

After a real `uninstall` (post V-2 fix) on a scratch `HOME` with the
`claude` client installed, the plugin cache directory tree under
`~/.claude/plugins/cache/token-optimizer-marketplace/token-optimizer/<version>/`
remained on disk (marked with an `.orphaned_at` file rather than deleted),
and `~/.claude/plugins/known_marketplaces.json` still listed the
`token-optimizer-marketplace` entry, even though
`~/.claude/plugins/installed_plugins.json` correctly showed no installed
plugins. The P1 section of the prior handoff explicitly calls out
"marketplace... residue" as something a real uninstall should leave behind
none of. This may be an intentional design choice (leaving client-owned
cache for the client's own GC rather than reaching into another tool's
cache format), or it may be a real gap — the existing test suite
(`test/installer/uninstall.test.ts`) does cover marketplace directory
removal for *some* layouts, so this needs a maintainer decision rather than
a same-session fix. Not changed this session.
