### Windows matrix (`windows-latest`) — three more failures, now fixed pending a real run

After the Ubuntu matrix went green, the maintainer reported the Windows leg
of the same job (`npm run test:security`) failing three tests, with the
exact raw TAP/assertion output (not a summary) supplied this time:

1. `package metadata and legal files consistently declare Apache-2.0`
   (`package-hygiene.test.ts`) — same SHA-256 mismatch pattern as the earlier
   Ubuntu LICENSE-hash failure, but for Windows: `LICENSE`/`NOTICE` had no
   `eol` directive in `.gitattributes`, so a Windows checkout with
   `core.autocrlf=true` (the Git for Windows default `windows-latest` uses)
   converts them to CRLF, changing the hash even though the committed
   content never changed. **Fixed:** pinned both files to `eol=lf` (matches
   both the root and `packages/installer` copies by basename).
2. `root and installer publish only their intentional package inventories`
   (`package-hygiene.test.ts`) — raw error showed `dryRunInventory`'s
   internal `assert.equal(result.status, 0, ...)` failing with `null !== 0`,
   i.e. the `npm pack --dry-run` child process never actually launched.
   **Root cause confirmed:** `npm` is a `.cmd` shim on Windows; `spawnSync`
   without `shell: true` fails to spawn it at all. **Fixed** here and in the
   identical (currently unreached by CI, but same latent bug) spawn in
   `release-preflight.test.ts`.
3. `scanner follows POSIX quote and escape rules and fails closed on
   unmatched quotes` (`command-boundary.test.ts`) — raw error showed
   `fs.existsSync(marker)` as `true` when `false` was expected, for the
   `printf x \| tee marker` case. **Root cause confirmed, and this one is a
   real security gap, not just a test artifact:** `src/command-policy.ts`'s
   `scanShellSyntax` models POSIX shell quoting only — backslash escapes the
   next metacharacter, and single quotes group their contents into an inert
   literal. `runner.ts` spawns with `shell: true` unconditionally, which
   resolves to **cmd.exe** on `win32`, and cmd.exe has neither of those
   rules: no backslash escape at all, and no single-quote grouping. The
   `printf x \| tee marker` case matches the observed failure exactly (the
   pipe is genuinely live under cmd.exe, so `tee marker` really ran).
   Reasoning through the single-quote case further surfaced a second,
   broader instance of the same gap that had no failing test: something like
   `printf 'x & del file'` is treated as one inert single-quoted blob under
   the POSIX model this scanner uses, but cmd.exe does not group via single
   quotes at all, so the `&` inside would be a live command separator on a
   real Windows execution. **Fixed** by making both allowances conditional
   on the resolved platform (defaulting to `process.platform`, overridable
   via a new optional `PolicyInput.platform` for tests); `win32` now fails
   closed (denies) on both cases instead of being silently wrong about what
   cmd.exe will actually do with the same command line. Added a unit test
   that injects `platform: 'win32'` into `evaluateCommand` directly, since
   no Windows environment was available this session to verify the actual
   cmd.exe integration end-to-end — only the scanner's logic is verified
   here, not a live cmd.exe run.

Commit `ec17317` contains all three fixes, verified with the full suite
(373/373) on macOS and in real `node:20`/`node:22` Linux Docker containers.

### Three more rounds on the Windows leg, each a genuinely distinct bug

Getting Windows green took three more iterations after `ec17317`, each a
real, distinct root cause rather than a repeat of the same mistake:

1. `df7537b` — the win32 command-scanner fix above had introduced its own
   regression: disabling single-quote *grouping* on win32 also silently
   disabled single-quote *unmatched-detection*, because `unmatchedQuote` was
   computed from whether `quote` was left "open," and win32 never entered
   quote mode at all anymore. `printf 'unterminated` came back `completed`
   instead of `blocked` on real Windows CI. Fixed by tracking single-quote
   parity as an independent counter on win32. Verified against the actual
   failing case by monkey-patching `process.platform` to `'win32'` and
   calling the real `runCommand()` — not just the unit-level scanner check.
2. `23d8d0e` — `npm run verify:generated` failed with `GENERATED_ASSET_DRIFT`
   across nearly every tracked generated file
   (`.claude-plugin/marketplace.json`, `packages/installer/assets/**`,
   `plugin/{claude,codex}/**`). Same category as the LICENSE/NOTICE fix, far
   more widespread: every generator writes LF unconditionally, but a Windows
   checkout with `core.autocrlf=true` converts the already-tracked copies to
   CRLF, so regenerating (LF) always diffed against the checkout (CRLF).
   Replaced the file-by-file LICENSE/NOTICE rule with a repo-wide
   `* text=auto eol=lf` default in `.gitattributes` (confirmed no actual
   binaries or `.bat`/`.ps1`/`.cmd` files exist in this repo that would need
   CRLF).
3. `0a8e9cb` — after (2), drift persisted for exactly two files:
   `packages/installer/assets/plugin/cursor/mcp-snippet.json` and the
   `opencode` equivalent. Not a line-ending issue this time:
   `scripts/generate-plugin-{cursor,opencode}.js` built an embedded
   `start.js` path with `path.join(installedServerDir, "start.js")`, where
   `installedServerDir` is a portable `"${HOME}/..."` placeholder string
   meant for the *target client* to interpret later, not a real filesystem
   path on the machine running the generator. `path.join` uses the *host*
   platform's separator, so generating on Windows CI produced
   `${HOME}\.cursor\...\start.js` (backslashes) against a
   forward-slash-committed file. Confirmed the exact mechanism with
   `require('path').win32.join(...)` locally (no live Windows host needed to
   prove this one) before fixing both generators to use plain forward-slash
   template interpolation instead.

**Maintainer confirmed on a real GitHub Actions run: Ubuntu, macOS, and
Windows are all green as of `0a8e9cb`.** This closes out P0 gate 4's CI
matrix requirement for the current `main` tip. All ten fix commits

the fix: reinstalled into a fresh scratch `HOME`, ran `uninstall --json`,

## V-7 (fixed) — `install`/`config`/`defaults` silently ignore `--home`, real-machine incident

While validating every provider (`gateway`, `byok`, `local`) against every
client (`claude`, `codex`, `antigravity`, `opencode`, `cursor`) using fake
secrets, every scratch-`HOME` test that relied on `--home` alone (without
also overriding the `HOME` environment variable) silently wrote to the
**real** developer machine's home directory instead. Root cause:
`packages/installer/bin/token-optimizer.js`'s `install`/`config`/`defaults`
command branch built its `options` object without `home: args.home` —
`status`, `doctor`, `repair`, `uninstall`, and `install --migrate` all
correctly included it; the plain `install`/`config`/`defaults` path did not.
`resolvePaths` inside `install-core.js` has always honored `options.home`
when present; it was simply never given one.

Real impact on this session's dev machine: `~/.claude/settings.json`,
`~/.codex/config.toml`, `~/.gemini/config/mcp_config.json`,
`~/.config/opencode/opencode.jsonc`, and `~/.cursor/mcp.json` were overwritten
with fake test values, and one fake credential was written to the real macOS
Keychain (`service: token-optimizer`, `account: gateway-token`). No other
real state was touched — every test used `--skip-launchctl`, so the real
`~/Library/LaunchAgents/com.softawarest.token-optimizer.env.plist` and live
`launchctl` environment (the actual source of truth for the working local-LLM
setup) were never touched and remained correct throughout, which is what made
recovery possible without guessing at the user's real prior values.

Recovery used the real installer itself — a normal `install --local` with
the known-good real endpoint/model, targeting the real `$HOME` — rather than
restoring whole files from the automatic `~/.token-optimizer-mcp/backups/`
snapshots, because those backups capture only "state before a write" and in
several cases predate unrelated legitimate real content (e.g. Antigravity's
backup had an empty `mcpServers: {}`, which would have deleted the real
`codegraph`/`notebooks`/`visualization` registrations added after that
backup). The installer's own managed-block writer only touches its own
section, so re-running it surgically repaired exactly the polluted keys and
left everything else untouched — confirmed by diffing every unrelated key
before/after. The stale fake-credential reference left behind in
`~/.token-optimizer/manifest.json`'s `credentials` array (metadata pointing
at the now-deleted Keychain entry) was cleared by hand since no CLI path
removes an orphaned array entry after the fact.

Fixed by adding `home: args.home` to the `install`/`config`/`defaults`
options object. Regression test added in `test/installer/cli.test.ts`
(`install/config/defaults honor --home even when it differs from the HOME
env var`) using a decoy `HOME` env value distinct from `--home`, so a
regression lands in a harmless scratch directory instead of a real machine —
confirmed failing pre-fix, passing post-fix.
