## 5. GitHub Actions matrix (P0 gate 4) ‚Äî confirmed broken; two of four fixed

Checked via the GitHub REST API (`GET /repos/.../actions/runs`,
`.../jobs`), not just the web UI summary (which under-reported the failure).

- Run for `94764e0` (branch `feat/hardenning-continue`): **conclusion:
  failure.**
  - `cross-platform (ubuntu-latest, 22, true)`: **failure**, at the
    `npm test` step (`Process completed with exit code 1`).
  - `cross-platform (ubuntu-latest, 20, true)`: cancelled (fail-fast side
    effect of the above).
  - `cross-platform (windows-latest, 22, false)`: cancelled (same).
  - `cross-platform (macos-latest, 22, false)`: success ‚Äî but this leg has
    `full: false` and does not run `npm test`, so it does not corroborate
    the suite passing.
- The maintainer supplied the actual failing-assertion text from the GitHub
  UI (this session could not read raw logs without authentication). Four
  distinct test failures on the Ubuntu legs:
  1. `generated server bundles include every runtime module required by
     llm.js` and 2. `generated plugin and installer server entrypoints load
     successfully` (`plugin-generators.test.ts`) ‚Äî both "antigravity missing
     providers.js" / "antigravity missing plugin index.js". **Root cause
     confirmed and fixed:** `.github/workflows/ci.yml` ran `npm test` before
     `npm run verify:generated`. `plugin/antigravity`, `plugin/opencode`,
     and `plugin/cursor` are gitignored build artifacts that only exist
     after `verify:generated` (via `build:installer` ‚Üí `build:plugin`)
     regenerates them; on a fresh checkout they don't exist yet when
     `npm test` runs, so the antigravity leg of the assertion loop (first in
     the `bundles` array) throws before opencode/cursor are even checked.
     Reproduced locally (deleted the three gitignored dirs, watched the
     exact same assertion fail) and confirmed the fix (reordered the two
     steps; regenerating the dirs first makes all four affected tests pass).
  3. `CLI emits one machine-readable stderr object and exits exactly one on
     policy failure` (`release-preflight.test.ts`) ‚Äî expected `TAG_REQUIRED`,
     got `TAG_POLICY_REJECTED` with tag `feat/hardenning-continue`. **Root
     cause confirmed and fixed:** the test spread `process.env` into the
     spawned CLI's env without scrubbing `GITHUB_REF_NAME`/`GITHUB_REF`,
     which GitHub Actions sets to the branch name; `scripts/release-preflight.js`
     falls back to `GITHUB_REF_NAME` when `RELEASE_TAG` is unset, so the
     branch name got validated as if it were a release tag. Fixed by
     deleting both vars from the spawned env; verified locally by setting
     `GITHUB_REF_NAME=feat/hardenning-continue` before running the test and
     confirming it still passes.
  4. `signal termination is distinct from timeout and retains the signal`
     (`tool-contracts.test.ts`) ‚Äî expected `executionStatus: 'terminated'`,
     got `'completed'`, on a child that sends itself `SIGTERM` via
     `process.kill(process.pid, 'SIGTERM')`. **Root cause confirmed and
     fixed**, using a real `node:20`/`node:22` Docker container (Docker was
     available on the dev machine after all) rather than guessing:
     `runCommand` spawns with `shell: true`. On macOS the default shell
     propagates a self-signaled grandchild's signal onto the wrapping
     shell's own `close` event, so `signal: 'SIGTERM'` came through
     directly ‚Äî that's why this was invisible locally. On Linux (dash/POSIX
     `sh`), the shell instead reports its *own* normal exit with
     `code: 143` (128+`SIGTERM`) and `signal: null`, so `runCommand` never
     detected real signal termination on Linux at all; every signaled
     command looked like `'completed'`. Reproduced the exact `code=143,
     signal=null` behavior in a bare Node script inside `node:22` before
     touching any product code. Fixed `src/runner.ts`'s `close` handler to
     derive the signal from the 128+n exit-code convention (the standard
     POSIX way a shell reports a signal-killed child) when Node's own
     `signal` field is `null`. This only adds detection for a case that
     was previously always misreported as `'completed'`; it does not loosen
     any existing guarantee. Verified `node --test` green on macOS and in
     both `node:20` and `node:22` Linux containers (full suite 372/372 on
     each). Regenerated `plugin/{claude,codex}` and
     `packages/installer/assets/plugin/*` via `npm run build:installer`
     since this changes shipped server behavior; no version bump, since
     `2.0.0-rc.6` is intentionally held for the deliberate P0-gate-7
     version-alignment step.
- Commits `9598fc6`, `8d73fe0`, and `557ea39` (pushed to `origin/main`; the

  real runners. **This gate is still not satisfied for the current

## Findings

### V-1 (informational) ‚Äî npm package name collision, not a release blocker

Root `package.json` (`token-optimizer-mcp`, unscoped, not `private`) matches
an unrelated published package (`ooples/token-optimizer-mcp`, currently
`2.17.0`). Confirmed this is cosmetic: `.github/workflows/publish-npm.yml`
only ever publishes `packages/installer` (`@softawarest/token-optimizer-installer`,
correctly scoped). No action taken per maintainer decision.

### V-2 (fixed) ‚Äî `repair --json` / `uninstall --json` silently no-opped real operations

`packages/installer/bin/token-optimizer.js` treated `args.json === true` as
equivalent to `args["dry-run"] === true` for the `repair`/`uninstall`
branch: `uninstall --json` (no `--dry-run`) printed a full operations plan,
exited 0, and reported `77` operations ‚Äî but **applied nothing**. Verified
by installing into a scratch `HOME`, running `uninstall --json`, and
confirming the manifest, LaunchAgent plist, and client caches were still on
disk afterward. `install`'s dry-run branch never had this bug (it only
checks `--dry-run`). No existing test exercised this path ‚Äî the only prior
`uninstall --json` test ran it on an already-uninstalled state, which
short-circuits before the buggy branch.

Fixed: the branch now only short-circuits on `--dry-run`; `--json` only
changes output formatting, matching `install`'s existing behavior. Verified
the fix: reinstalled into a fresh scratch `HOME`, ran `uninstall --json`,
confirmed the manifest and plist were actually removed. Added a regression
test (`test/installer/cli.test.ts`, "uninstall --json without --dry-run
actually applies, not just previews"). Updated
`packages/installer/README.md` to describe `--json` as an output-format
flag, not a preview flag. Full suite: 372/372 passing after the fix.

### V-3 (informational) ‚Äî local LLM gateway unreachable in this environment

`check_local_llm_health` and `run_test_verdict` both reported
`401 Unauthorized` from the configured gateway
(`https://llm-proxy.lnf.gr/v1`). Command execution and exit-code handling
remained correct and authoritative regardless (by design ‚Äî see
`docs/security/threat-model.md` and this repo's `CLAUDE.md`), and the tool
degraded to `"uncertain"` rather than fabricating a confident verdict, which
matches the documented conservative-fallback contract. Flagging because this
means no real LLM-triage path was exercised this session ‚Äî only the
deterministic command path. Gateway credential health is outside this
session's authority to fix.

### V-4 (open, not fixed this session) ‚Äî Python auto-detection misses pyproject.toml-only projects

`src/detector.ts`'s Python branch only fires on `pytest.ini`, `conftest.py`,
`requirements.txt`, or `Pipfile`. A `pyproject.toml`-only project (Poetry,
uv, hatch, PEP 621 ‚Äî the current standard for new Python projects) is never
detected; `detectCommands` returns `[]`, and `run_test_verdict` reports "No
tests or build tools detected" even with a real test suite present.
Confirmed with a real fixture workspace and the actual MCP tool call.
Flagged as a background task (not implemented in this session) rather than
fixed inline, since it changes detection behavior for a whole ecosystem and
deserves its own test coverage and doc update per this repo's `CLAUDE.md`.

### V-5 (open, not fixed this session) ‚Äî possible marketplace-cache residue after uninstall

After a real `uninstall` (post V-2 fix) on a scratch `HOME` with the
`claude` client installed, the plugin cache directory tree under
`~/.claude/plugins/cache/token-optimizer-marketplace/token-optimizer/<version>/`
remained on disk (marked with an `.orphaned_at` file rather than deleted),
and `~/.claude/plugins/known_marketplaces.json` still listed the
`token-optimizer-marketplace` entry, even though
`~/.claude/plugins/installed_plugins.json` correctly showed no installed
plugins. The P1 section of the prior handoff explicitly calls out
"marketplace... residue" as something a real uninstall should leave behind
none of. This may be an intentional design choice (leaving client-owned
cache for the client's own GC rather than reaching into another tool's
cache format), or it may be a real gap ‚Äî the existing test suite
(`test/installer/uninstall.test.ts`) does cover marketplace directory
removal for *some* layouts, so this needs a maintainer decision rather than
a same-session fix. Not changed this session.
